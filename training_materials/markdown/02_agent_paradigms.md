# 三种经典智能体构建方式

这部分对应 `hello-agents` 第四章，但这里的组织方式更偏培训讲义，而不是原书行文。

## 1. ReAct

### 一句话定义

ReAct 是“边想边做”的 Agent Loop：模型先思考，再调用工具，再根据工具结果继续思考。

### 它更侧重什么

- 侧重实时反馈
- 侧重工具联动
- 侧重在不确定环境里边走边修正

### 它最适合什么任务

- 搜索问答
- API 调用
- 需要实时外部信息的任务
- 路径无法提前完全确定的探索型任务

### 最小实现要点

1. 让模型按固定格式输出 `Thought` 和 `Action`
2. 解析 `Action`
3. 真正执行工具
4. 把 `Observation` 追加回上下文
5. 继续下一轮，直到得到最终答案

### 培训时建议强调

- ReAct 的重点不在“聪明”，而在“闭环”
- 如果工具结果不回流，ReAct 就只剩下“假装会调用工具”
- 它通常是最适合拿来做第一版教学 Demo 的范式

## 2. Plan-and-Solve

### 一句话定义

Plan-and-Solve 是“先定路线，再按路线执行”的两阶段范式。

### 它更侧重什么

- 侧重前置规划
- 侧重步骤稳定
- 侧重把复杂任务拆成更容易控制的子任务

### 它最适合什么任务

- 多步推理
- 结构化生成
- 需要明确阶段边界的流程型任务

### 最小实现要点

1. 单独做一个 `Planner`
2. 先让模型输出结构化计划
3. 再用 `Executor` 一步步执行计划
4. 在执行阶段显式传入历史步骤结果

### 培训时建议强调

- 这类范式不是为了更灵活，而是为了更稳
- 它的优点来自“规划阶段”和“执行阶段”的职责分离
- 它比 ReAct 更像“施工图”而不是“侦探办案”

## 3. Reflection

### 一句话定义

Reflection 是“先出第一版，再自我反思并迭代优化”的范式。

### 它更侧重什么

- 侧重质量提升
- 侧重内部纠错
- 侧重把“结果差不多”提升到“结果更可靠”

### 它最适合什么任务

- 代码生成与优化
- 长文本打磨
- 报告优化
- 重要决策建议

### 最小实现要点

1. 先生成初稿
2. 再生成批判性反馈
3. 根据反馈重写
4. 通过记忆模块保存每一轮轨迹
5. 到达停止条件后返回最终结果

### 培训时建议强调

- Reflection 的本质是“用额外成本换更高质量”
- 它不是默认最优，而是要看任务值不值得多轮调用
- 它适合作为高价值场景的“二次质量门”

## 三种范式怎么选

### 如果任务路径不确定

优先考虑 ReAct。

### 如果任务步骤很清晰

优先考虑 Plan-and-Solve。

### 如果结果质量是底线

优先考虑 Reflection。

### 如果是真实业务系统

大多数时候不会单独使用某一种，而是组合：

- `ReAct + Reflection`
- `Plan-and-Solve + ReAct`
- `Plan-and-Solve + Reflection`

这也是培训时可以顺带引导团队理解的重点：  
**范式不是互斥选项，而是可组合的工作流设计单元。**
