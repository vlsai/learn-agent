# 三种经典智能体构建方式

这份讲义只围绕一个统一任务展开：

> 生成一个单文件贪吃蛇网页，包含移动、计分、失败和重新开始。

统一任务的意义在于，同样的目标、不同的工作流，差异会更容易被看清。

## 1. ReAct

### 一句话定义

ReAct 是“先思考，再行动，再根据反馈继续思考”的闭环。

### 它在这个 demo 里更侧重什么

- 边做边查
- 工具调用
- 外部反馈回流

### 在贪吃蛇任务里的典型行为

- 查询本地需求说明
- 查询玩法和 UI 约束
- 必要时计算网格尺寸或速度参数
- 最后生成完整网页代码

### 最小实现要点

1. 约束模型输出 `Thought` 和 `Action`
2. 程序解析 `Action`
3. 执行本地工具
4. 把 `Observation` 写回上下文
5. 继续循环直到产出最终答案

### 培训时建议强调

- ReAct 的关键是闭环
- 工具结果如果没有回流，ReAct 就退化成“假装会用工具”
- 它最适合拿来讲 Agent Loop 的基本骨架

### 配套代码

- `code/paradigms/react_demo.py`

## 2. Plan-and-Solve

### 一句话定义

Plan-and-Solve 是“先拆解，再执行，再汇总”的两阶段工作流。

### 它在这个 demo 里更侧重什么

- 先把任务拆清楚
- 再按步骤生成中间结果
- 最后统一合成完整文件

### 在贪吃蛇任务里的典型拆法

- 页面结构
- 游戏状态
- 移动与碰撞逻辑
- 重新开始和交互细节
- 代码合并与检查

### 最小实现要点

1. 单独准备一个 `Planner`
2. 让模型先输出结构化计划
3. 用 `Executor` 一步一步执行
4. 再由 `Composer` 合成最终 HTML

### 培训时建议强调

- 这个范式不是更灵活，而是更可控
- 一旦任务天然分阶段，它通常比直接一把生成更稳
- 它很适合结构化生成和流程型任务

### 配套代码

- `code/paradigms/plan_and_solve_demo.py`

## 3. Reflection

### 一句话定义

Reflection 是“先产出第一版，再审查，再重写”的迭代工作流。

### 它在这个 demo 里更侧重什么

- 结果质量
- 发现隐藏问题
- 利用反馈推动第二版优化

### 在贪吃蛇任务里的典型检查点

- 操作是否顺手
- 失败条件是否完整
- 重开是否真正重置状态
- 注释是否足够适合培训演示

### 最小实现要点

1. 先生成第一版网页代码
2. 再从玩法完整性、代码清晰度、潜在 bug 风险几个角度给反馈
3. 根据反馈重写
4. 保存每轮轨迹，便于回看和讲解

### 培训时建议强调

- Reflection 不是默认必须开启
- 它的本质是多花几轮调用，换更高质量
- 对代码、报告、决策类结果，它很常见

### 配套代码

- `code/paradigms/reflection_demo.py`

## 三种范式怎么选

### 如果任务在做的过程中还要不断补信息

优先考虑 ReAct。

### 如果任务可以先拆成稳定步骤

优先考虑 Plan-and-Solve。

### 如果任务已经有初稿，但你担心质量

优先考虑 Reflection。

## 一条更接近真实项目的结论

真实业务里，三者经常是组合而不是互斥：

- 先用 `Plan-and-Solve` 拆任务
- 局部执行时用 `ReAct` 查资料
- 最终交付前再用 `Reflection` 做质量提升

所以培训时最好把它们理解成三种工作流积木。
